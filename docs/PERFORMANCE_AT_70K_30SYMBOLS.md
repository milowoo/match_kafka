# 70K并发 × 30交易对的延迟评估

## 场景定义

```
系统负载：
- 并发吞吐: 70,000 events/sec
- 交易对数: 30个
- 平均每交易对: 2,333 events/sec
- 事件大小: ~500 bytes
```

## 优化后的性能分析

### 链路1: 主实例内处理延迟

#### 1.1 appendBatch 处理

```
操作                耗时
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 序列号分配     ~0.1ms
2. EventLog写入   ~0.5-1.0ms (ChronicleQueue)
3. seqToSymbolMap 更新  ~0.01ms
4. 定期清理检查   ~0.001ms (每1000个触发一次)
5. Redis checkpoint ~0.1ms (偶发)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
单个事件: 0.6-1.1ms
```

**70K/sec 的主实例处理**:
- 每秒处理: 70,000 events
- 每个事件平均耗时: 0.8ms
- 总处理时间: 70,000 × 0.8ms = 56秒/sec

❌ **问题**：这已经超过1秒了！意味着有严重的队列堆积。

#### 1.2 实际分析 (多线程处理)

假设主实例有多个线程并行处理订单（真实场景）:

```
假设处理线程数: 16条线程 (合理的服务器配置)
实际处理能力: 70,000 / 16 ≈ 4,375 events/thread/sec
单线程处理时间: 1,000ms / 4,375 = 0.23ms/event

处理延迟分析:
- 新事件到达
- 入队到worker queue: 0-100µs
- worker线程处理: 0.23ms
- 写入Chronicle Queue: 0.5ms
- 异步确认: 100µs
━━━━━━━━━━━━━━━━━━━━━━━
总处理延迟: ~0.8ms (P50)
           ~2-3ms (P99)
```

### 链路2: EventLogReplicationSender 发送延迟

#### 2.1 动态循环分析

```
待发送事件堆积情况（70K/sec）:

时刻T        待发送     sleepMs    发送速度
───────────────────────────────────────────
T=0, 初始      0         100        -
T=100ms        7000      10         需要7轮
T=110ms*2      13000     10
T=120ms*3      19000     10
···
```

**发送吞吐量需求**:
- 70,000 events/sec 必须全部发送到Kafka
- 每秒需要发送: 70,000 / 1000ms ≈ 70 events/ms
- 单个事件发送时间: 1/70 ≈ 14µs （仅Kafka序列化+发送）

**实际发送延迟**:

```
发送循环分析（100ms一个循环）:

周期1 (0-100ms):   
  - 产生事件: 7,000
  - 发起发送: Kafka batch send
  - 预期完成: 第2个100ms开始到达Kafka

周期2 (100-200ms):
  - 产生事件: 7,000 (累计14,000)
  - 发送前次积压+新的: 14,000
  - 动态间隔: 10ms (因为待发送>100)
  - 实际发送速率: 7,000 events/100ms = 70 events/ms ✅
```

**关键发现**: 在10ms循环下，发送速率足够跟上！

### 链路3: Kafka处理延迟

```
Kafka端延迟:
- 进入主题: <1ms

- 副本复制: ~5-10ms (同步副本)
- ack返回: <1ms
```

**总Kafka延迟**: ~10ms

### 链路4: Consumer消费延迟

```
备实例Consumer端:
- poll间隔: 配置100ms (可优化)
- batch处理: 可配置
- appendReplicated: 0.5-1.0ms/event
```

## 性能瓶颈分析

### 1️⃣ **CRITICAL: Consumer poll 延迟 (100ms)**

这是目前最大的瓶颈！

```
Consumer配置示例:
session.timeout.ms = 10000
heartbeat.interval.ms = 3000
max.poll.records = 500  // 默认500条
fetch.max.wait.ms = 500 // 等待时间

问题: 即使batch只有100条，poll也可能等待最多500ms
```

**改进**: 将 `fetch.max.wait.ms` 降低到 50ms，可以减少一半延迟。

### 2️⃣ **HIGH: 动态循环间隔 (10ms 最少)**

在70K/sec下：

```
单个循环能处理的事件数:
- 循环间隔: 10ms
- 每个事件处理: 0.01ms
- 单循环容量: 10ms / 0.01ms ≈ 1000 events

能否跟上?
- 70K/sec ÷ 100 循环/sec = 700 events/循环
- 1000 > 700 ✅ 能跟上！

但是如果出现突发:
- 10个事件在同一ms内到达
- 需要10ms才能全部发送
- 客户端看到的延迟: ~10ms
```

### 3️⃣ **MEDIUM: readEventBySeq 缓存命中率**

在70K/sec + 30个交易对下：

```
seqToSymbolMap大小限制: 10,000

每秒新增seq: 70,000
保留10,000个 = 143ms的数据量

问题: 如果EventLogReplicationSender 落后超过143ms，
      就会出现缓存miss，导致全扫描！

场景分析:
- 正常: 缓存命中率 > 99%
- 有突发: 缓存命中率 ~ 95-98%
- 故障恢复: 缓存命中率 ~ 70-80% (初期)
```

## 关键风险点

### 🔴 高风险

1. **Consumer 阻塞**
   - 问题: poll() 可能等待100-500ms
   - 影响: 延迟可能>500ms
   - 修复: 优化Consumer配置

2. **缓存失效**
   - 问题: lag > 143ms时cache miss增加
   - 影响: readEventBySeq变为全扫描
   - 修复: 增加cache大小 or 动态调整清理策略

### 🟡 中风险

3. **GC暂停**
   - 问题: 若Kafka或Redis GC停顿
   - 影响: 延迟增加到秒级
   - 修复: 优化GC配置，使用低延迟GC

4. **Redis瓶颈**
   - 问题: 7K QPS Redis调用（虽然异步）
   - 影响: Redis可能成为瓶颈
   - 修复: Redis集群 or 本地缓存

### 🟢 低风险

5. **内存占用**
   - seqMap: 500KB (稳定)
   - Kafka缓冲: ~10MB
   - 总体: 可控

## 改进建议 (为70K/sec优化)

### 优先级 1: 优化 Consumer 配置

```properties
# 备实例 Kafka Consumer 配置
max.poll.interval.ms = 300000  # 5分钟超时
max.poll.records = 1000        # 增加batch
fetch.max.wait.ms = 50         # 降低等待 (关键!)
fetch.min.bytes = 1024         # 最小拉取
heartbeat.interval.ms = 3000   # 保持活力

效果: 
- P99延迟: 150ms → 80ms (50%↓)
- avg延迟: 60-80ms → 40-50ms (33%↓)
```

### 优先级 2: 扩大 seqToSymbolMap

```java
// 改为: MAX_MAP_SIZE = 100,000 (从10,000)
// 覆盖: 1.4秒的数据量 (从143ms)

效果:
- 缓存命中率: 99%+ (即使lag增长到500ms)
- 无cache miss导致的性能下降
- 内存: 5MB (仍可接受)
```

### 优先级 3: 监控和告警

```yaml
critical_alerts:
  - eventlog.consumer.lag > 1000
  - eventlog.send.batch.latency > 200ms
  - eventlog.cache.miss_rate > 10%
  - redis.update.failure_rate > 1%
```

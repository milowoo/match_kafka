# 从实例切换到主实例流程分析

## 触发方式

| 路径 | 触发方式 | 说明 |
|------|---------|------|
| 计划内切换 | 运维调用 `POST /ops/ha/activate` | 正常升主 |
| OOM 自动恢复 | JVM 重启后 `autoRecoverIfWasPrimary()` | Redis 有 PRIMARY 标志时自动触发 |

---

## 状态转换

```
STANDBY → ACTIVATING → PRIMARY
失败时回滚: ACTIVATING → STANDBY
```

ACTIVATING 期间：
- `isActive() = false`，`PlaceOrderCommand` / `CancelOrderCommand` 拒绝所有新订单
- `EventLogReplicationConsumer` 继续消费 `MATCHING_EVENTLOG_SYNC`（始终运行）

---

## 完整步骤

```
运维: POST /ops/ha/activate
         ↓
HAService.activate()
  [状态检查] role.compareAndSet(STANDBY, ACTIVATING)
  Step 1: recoverFromBackup()         ← 数据恢复
  Step 2: eventLog.switchAllToPrimary() ← EventLog 切换为可写
  Step 3: startConsumers()            ← 启动 Kafka 消费
  Step 4: role.set(PRIMARY)           ← 更新状态
           leaderElection.activate()
           registerHeartbeat()
           persistRole("PRIMARY")     ← 写入 Redis PRIMARY 标志
  失败时: 回滚到 STANDBY
```

---

## 各步骤详解

### [状态检查]

```java
if (!role.compareAndSet(Role.STANDBY, Role.ACTIVATING)) {
    return false;  // 非 STANDBY 状态直接拒绝
}
```

- 原子 CAS，防止并发重入
- 只有 STANDBY 才能进入 ACTIVATING

---

### Step 1: recoverFromBackup()

```
FastRecoveryService.recoverAllSymbols(symbols):
  for each symbol（顺序恢复）:
    1a. loadLatestSnapshot(symbol)
        → snapshotService.readSnapshotFromRedis(symbol)
        → engine.loadOrders(snapshot.getAllOrders())
        → 恢复 OrderBook 到 seq=M

    1b. replayWALEvents(symbol, snapshotSeq)
        → eventLog.readEvents(symbol, snapshotSeq)
        → for each event: replayEvent(engine, event)
            → engine.addOrderFromDTO(addedOrders)
            → engine.cancelOrder(cancelledOrderIds)
        → 增量回放 seq=M+1 到 seq=N

    1c. verifyRecovery(symbol)
        → 检查买卖盘价格排序完整性

    1d. orderBookManager.markAsLoaded(symbol)
        → 标记为已加载，Disruptor 可以处理该 symbol 的订单

EventLogLoader.loadFromEventLog（在 FastRecoveryService 内部调用）:
    1e. 幂等缓存预热（修复A）
        → 当前挂单 → markLocal
        → lastSentSeq 到 seq=N 窗口内已成交订单 → markLocal

    1f. 补发增量 EventLog 给从实例（修复B）
        → replicateEvent(seq=M+1 到 seq=N)
        → 确保从实例数据追上
```

**⚠️ 隐患1**：`recoverAllSymbols` 是**顺序**恢复，10 个交易对串行执行，每个约 10~40ms，总计 100~400ms。期间 ACTIVATING 状态不处理订单，有短暂服务中断。

**⚠️ 隐患2**：`replayWALEvents` 里用的是 `eventLog.readEvents(symbol, fromSeq)`，但 `FastRecoveryService` 注入的是 `ChronicleQueueEventLog`，而 `EventLogLoader` 里的幂等预热和补发逻辑是独立的，两者之间存在**重复回放**风险：`FastRecoveryService.replayWALEvents` 和 `EventLogLoader.loadFromEventLog` 都会回放 EventLog，可能导致同一订单被 `addOrderFromDTO` 两次（`addOrderFromDTO` 有幂等保护，跳过已存在订单，不影响正确性，但有冗余）。

---

### Step 2: eventLog.switchAllToPrimary()

```
ChronicleQueueHAOperations.switchAllToPrimary():
  queueFactory.switchAllToPrimary()
  for each symbol:
    components.haManager.switchToPrimary()
      → ChronicleQueueHAManager.switchToPrimary()
          → performConsistencyCheck()    ← 检查本地 EventLog seq 一致性
          → queueManager.switchToPrimary() ← Chronicle Queue 切换为可写
          → recoverFromQueue()           ← 从本地 EventLog 恢复 globalSeq
```

**⚠️ 隐患3**：`switchAllToPrimary` 在 Step 1 之后执行，但 `appendBatch` 里有 `!isPrimary()` 判断。Step 1 恢复完成后，如果 Disruptor 里已有消息入队（Step 3 启动 Kafka 消费之前不应该有，但理论上存在竞态），这些消息调用 `appendBatch` 时 Chronicle Queue 还未切换为主模式，会直接返回不写入。

**实际上不会发生**：Step 3 启动 Kafka 消费在 Step 2 之后，Disruptor 里不会有消息，此隐患可忽略。

---

### Step 3: startConsumers()

```
KafkaConsumerStartupService.startConsumers():
  shouldBeRunning.set(true)              ← 先设置，防止消息到达时被丢弃
  kafkaListenerRegistry.start()          ← 启动 ReceiverMq（TRADE_TOP_SYMBOL）
  shouldBeRunning.set(true)
  verifyStartup()                        ← 等待 2 秒验证启动结果
    → kafkaListenerRegistry.isRunning()
    → 所有 container.isRunning()
    → kafkaListenerMonitor 状态检查
```

- 启动后 `ReceiverMq.onMessage` 里 `shouldBeRunning=true`，开始处理新订单
- `EventLogReplicationConsumer` 不受影响（独立 `replicationContainerFactory`）
- Kafka 从上次 committed offset 继续消费，未提交的订单会被重消费

**⚠️ 隐患4**：`verifyStartup` 固定等待 2 秒，增加了 activate 总时间。可改为轮询检测，实际就绪后立即返回。

---

### Step 4: 更新状态

```java
role.set(Role.PRIMARY)           // isActive() = true，开始处理订单
leaderElection.activate()        // InstanceLeaderElection: STANDBY → ACTIVE
registerHeartbeat()              // 写 Redis heartbeat key
persistRole("PRIMARY")           // 写 Redis role key，OOM 重启后可自动恢复
```

**⚠️ 隐患5**：`role.set(PRIMARY)` 和 `persistRole("PRIMARY")` 之间有时间窗口。如果在这个窗口内 OOM，Redis 里没有 PRIMARY 标志，重启后不会自动恢复，需要运维手动 activate。

---

### 失败回滚

```java
kafkaConsumerStartupService.stopConsumers()  // 停止已启动的 Kafka 消费
eventLog.switchAllToStandby()                // 切回只读模式
role.set(Role.STANDBY)                       // 回滚状态
leaderElection.completeDeactivate()
```

**⚠️ 隐患6**：回滚时如果 `stopConsumers` 失败，Kafka 消费者仍在运行，但 `role=STANDBY`，`isActive()=false`，`ReceiverMq.onMessage` 里 `shouldBeRunning=false` 会直接 ack 丢弃消息。消息不会被处理，但 offset 会被提交，**导致消息丢失**。

---

## 完整时序图

```
STANDBY
  ↓ [CAS: STANDBY → ACTIVATING]
ACTIVATING
  ↓ Step 1: recoverFromBackup()
    ├── Redis Snapshot → OrderBook(seq=M)        ~5-20ms
    ├── 本地 EventLog 增量回放(seq=M+1~N)        ~1-10ms
    ├── 幂等缓存预热                              ~1ms
    └── 补发增量 EventLog 给从实例               ~5-20ms
  ↓ Step 2: switchAllToPrimary()
    └── Chronicle Queue 切换为可写               ~1-5ms
  ↓ Step 3: startConsumers()
    └── Kafka 消费者启动 + verifyStartup(2s)     ~2-3s
  ↓ Step 4: role=PRIMARY
    └── 开始处理订单
PRIMARY
```

**总 activate 时间：约 3~5 秒**（主要瓶颈是 Kafka 消费者启动的 2 秒等待）

## 隐患汇总

| 隐患 | 严重程度 | 说明 |
|------|---------|------|
| 隐患1：顺序恢复导致服务中断 | 低 | 100~400ms，可接受 |
| 隐患2：FastRecoveryService 和 EventLogLoader 重复回放 | 低 | 有幂等保护，不影响正确性 |
| 隐患3：Step 2 前 appendBatch 不写入 | 无 | Step 3 在 Step 2 后，实际不会发生 |
| 隐患4：verifyStartup 固定等待 2 秒 | 低 | 增加 activate 时间，可优化 |
| 隐患5：role=PRIMARY 和 persistRole 之间的 OOM 窗口 | 低 | 需运维手动恢复，概率极低 |
| 隐患6：回滚时 stopConsumers 失败导致消息丢失 | 高 | offset 已提交但消息未处理 |

---

## 隐患6 详细分析（高优先级）

**触发条件**：
```
activate() 执行到 Step 3 startConsumers() 成功
  ↓ Step 4 某处抛异常（如 persistRole Redis 超时）
  ↓ 进入 catch 块回滚
  ↓ stopConsumers() 失败（Kafka 连接异常）
  ↓ Kafka 消费者仍在运行
  ↓ role=STANDBY，shouldBeRunning=false
  ↓ ReceiverMq.onMessage: shouldBeRunning=false → ack 丢弃
  ↓ 消息 offset 提交，但未处理 ← 数据丢失
```

**修复建议**：回滚时 `stopConsumers` 失败，不应该 ack 消息。需要在 `ReceiverMq.onMessage` 里增加双重判断：

```java
// 当前只判断 shouldBeRunning
if (!kafkaConsumerStartupService.shouldBeRunning()) {
    ack.acknowledge();  // ← 危险：直接 ack 丢弃
    return;
}

// 修复：同时判断 role
if (!kafkaConsumerStartupService.shouldBeRunning() || !haService.isActive()) {
    // 不 ack，让 Kafka 重投
    return;
}
```

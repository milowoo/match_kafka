# 整个框架的核心理解

## 分层架构
┌─────────────────────────────────────────┐
│   Controller Layer (REST API)           │
│   - TradeController: 下单/撤单接口      │
│   - HAController: 切换接口              │
│   - OpsController: 运维接口             │
└──────────────│──────────────────────────┘
↓
┌─────────────────────────────────────────┐
│   Command Layer (命令处理)               │
│   - PlaceOrderCommand: 下单处理          │
│   - CancelOrderCommand: 撤单处理         │
│   - MatchExecutor: 撮合循环执行          │
└──────────────│──────────────────────────┘
↓
┌─────────────────────────────────────────┐
│   Disruptor Layer (无锁并发)            │
│   - MatchEventHandler: 事件处理器        │
│   - MatchEvent: 事件对象                │
│   - 单RingBuffer + 单消费线程           │
└──────────────│──────────────────────────┘
↓
┌─────────────────────────────────────────┐
│   Engine Layer (核心撮合引擎)           │
│   - MatchEngine: 订单簿管理             │
│   - PriceLevelBook: 跳表价格簿           │
│   - OrderList: 链表订单链               │
│   - CompactOrderBookEntry: 订单对象      │
└──────────────│──────────────────────────┘
↓
┌─────────────────────────────────────────┐
│   Service Layer (业务服务)              │
│   - OrderBookService: 订单簿服务        │
│   - EventLog: 事件日志                  │
│   - ChronicleQueueEventLog: 持久化      │
│   - HAService: 高可用管理               │
│   - ResultOutboxService: 结果投箱        │
│   - IdempotentService: 幂等检查         │
│   - SnapshotService: 快照服务           │
└─────────────────────────────────────────┘

## 核心数据结构
组件                    功能                            特点
MatchEngine          交易对订单簿管理    多个实例（per-symbol）存在，
                                    全部由Disruptor线程管理

PriceLevelBook     价格级别管理        ConcurrentSkipListMap，
                                      支持并发读取深度数据
OrderList           同价订单链表       侵入式链表，O(1)删除，支持VIP插队
CompactOrderBookEntry  订单对象   对象池复用，long价格/数量表示，支持冰山单
EntryPool           对象池                   20000容量预分配，减少GC压力

## 核心流程路径

### 1.下单流程
```
Kafka → Disruptor RingBuffer → MatchEventHandler
  ↓ (Phase 1: 幂等性检查)
  ├→ PlaceOrderCommand 执行
  ├→ 撮合循环 (MatchExecutor)
  ├→ 构建EventLog.Event
  ↓ (Phase 2: EventLog持久化 - 关键路径)
  ├→ ChronicleQueueEventLog.appendBatch()
  ├→ 事务性写入 + seq回滚
  ├→ ack Kafka
  ↓ (Phase 3: asyncIoExecutor异步处理)
  ├→ EventLog复制 (Kafka MATCHING_EVENTLOG_SYNC)
  ├→ Redis幂等标记
  ├→ 发送成交结果
  ├→ 深度推送
  └→ 定期Snapshot (60s)

```
### 2.撤单流程
```
Kafka → Disruptor → MatchEventHandler
  ↓
CancelOrderCommand 执行
  ├→ 从orderIndex快速查询 (O(1))
  ├→ 从OrderList移除 (O(1))
  ├→ 构建EventLog
  ↓
ChronicleQueueEventLog persist + ack
  ↓
asyncIoExecutor异步操作

```

### 3.主从复制流程
```
主实例写入 → Kafka MATCHING_EVENTLOG_SYNC topic
          ↓
从实例读取 → EventLogReplicationConsumer
         ↓
过滤检查 (不是自己发出的，当前是STANDBY)
         ↓
EventLog.appendReplicated() 写入
         ↓
保持主从seq一致

```

### 4.关键技术点
```
技术                        实现                  价值
无锁设计               单Disruptor单线程消费        零锁竞争，微秒级延迟
价格计算               BigDecimal → long           3.57倍性能提升
对象复用               EntryPool                 减少GC，内存优化
并发读                ConcurrentSkipListMap       深度数据可并发读取
事务性写入             seq预分配+回滚              保证EventLog原子性
三阶段流水线           关键路径分离                 Phase1-2同步，Phase3异步
快照恢复            Redis Snapshot + EventLog增量回放  10-30ms故障转移

```

### 5.OOM场景下的数据保护机制
```
Layer 1: EventLog事务性写入 (单机本地）
Layer 2: 幂等控制 (本地LRU+Redis)
Layer 3: Kafka异步复制 (min.insync.replicas=2)
Layer 4: Redis Snapshot (定期60秒)
Layer 5: ResultOutboxService重试队列
Layer 6: BatchFailureRecoveryService恢复
Layer 7: DeadLetterService死信队列


```

###  6. 高可用状态机
```

STANDBY ──activate──→ ACTIVATING
  ↑                      │
  │                    (恢复Snapshot+回放EventLog)
  │                      │
  └─activate──→ PRIMARY ──deactivate──→ DEACTIVATING
                           ↑               │
                           └──create final Snapshot─→ STANDBY

```

# 主实例切换到从实例流程分析

## 状态机设计

引入 `ACTIVATING` / `DEACTIVATING` 中间状态，避免切换过程中的状态不一致：

```
STANDBY ──activate──→ ACTIVATING ──成功──→ PRIMARY
                           └──失败──→ STANDBY（回滚）

PRIMARY ──deactivate──→ DEACTIVATING ──成功──→ STANDBY
                              └──失败──→ PRIMARY（回滚）
```

| 状态 | 语义 | isActive | 处理订单 |
|------|------|---------|---------|
| STANDBY | 从实例，待命 | false | ❌ |
| ACTIVATING | 正在激活，恢复数据中 | false | ❌ |
| PRIMARY | 主实例，正常服务 | true | ✅ |
| DEACTIVATING | 正在降级，停止服务中 | false | ❌ |

---

## 触发方式

| 路径 | 触发方式 | 说明 |
|------|---------|------|
| 计划内切换 | 运维调用 `POST /ops/ha/deactivate` | 正常降级，有序停止 |
| 进程退出 | JVM 收到 SIGTERM（systemd stop） | 触发 GracefulShutdown |
| OOM 重启 | JVM OOM 后自动重启 | 触发 autoRecoverIfWasPrimary |

---

## 路径一：计划内切换（deactivate）

### 状态转换

```
PRIMARY → DEACTIVATING → STANDBY
失败时回滚: DEACTIVATING → PRIMARY
```

### 完整步骤

```
运维: POST /ops/ha/deactivate
         ↓
HAService.deactivate()
  [状态检查] role.compareAndSet(PRIMARY, DEACTIVATING)
  Step 1: stopConsumers()          ← 停止 Kafka 消费
  Step 2: createFinalSnapshot()    ← 创建最终 Snapshot 写 Redis
  Step 3: switchAllToStandby()     ← EventLog 切换为只读
  Step 4: role.set(STANDBY)        ← 更新状态
  Step 5: unregisterHeartbeat()    ← 清除心跳
  Step 6: persistRole("STANDBY")   ← 清除 Redis PRIMARY 标志
  失败时: 回滚到 PRIMARY
```

### 各步骤详解

**[状态检查]**
```java
if (!role.compareAndSet(Role.PRIMARY, Role.DEACTIVATING)) {
    return false;  // 非 PRIMARY 状态直接拒绝，防止并发重入
}
```
- 原子 CAS 操作，确保只有 PRIMARY 才能进入 DEACTIVATING
- DEACTIVATING 期间 `isActive()=false`，`PlaceOrderCommand` 拒绝所有新订单

**Step 1: stopConsumers()**
```
KafkaConsumerStartupService.stopConsumers():
  shouldBeRunning.compareAndSet(true, false)
  kafkaListenerMonitor.setShouldBeRunning(false)
  kafkaListenerRegistry.stop()    ← 停止 Kafka 消费，释放 partition
  verifyStop()                    ← 验证停止结果
```
- 停止后 `ReceiverMq.onMessage` 里 `shouldBeRunning=false`，新消息直接 ack 丢弃
- **注意**：Disruptor 里可能还有存量消息在处理，Snapshot 不一定是最终状态，但有 EventLog 兜底，不影响数据安全

**Step 2: createFinalSnapshot()**
```
遍历所有 MatchEngine:
  snapshotService.snapshot(symbolId, engine)
    → engine.getBuyBook().allOrders() + getSellBook().allOrders()
    → writeSnapshotToRedis()    ← 写 Redis（Protobuf 序列化）
    → writeLastSentSeq()        ← 更新 lastSentSeq
```
- 某个 symbol 写失败时记录告警，继续处理其他 symbol
- 从实例 activate 时对缺失 Snapshot 的 symbol 做全量 EventLog 回放兜底

**Step 3: switchAllToStandby()**
```
ChronicleQueueHAOperations.switchAllToStandby():
  queueFactory.switchAllToStandby()
  for each symbol:
    components.haManager.switchToStandby()
      → ChronicleQueueManager.switchToStandby()
        → appender 关闭，Chronicle Queue 切换为只读模式
```
- 切换后 `appendBatch` 里 `!isPrimary()` 直接返回，不再写 EventLog

**Step 4: 更新状态**
```java
role.set(Role.STANDBY)
leaderElection.startDeactivate()
leaderElection.completeDeactivate()
```

**Step 5 & 6: 清理标志**
```
unregisterHeartbeat()    ← 删除 Redis heartbeat key
persistRole("STANDBY")  ← 删除 Redis role key，防止重启后误自动激活
```

**失败回滚**
```java
// 任意步骤失败时：
kafkaConsumerStartupService.startConsumers()  // 重新启动 Kafka 消费
eventLog.switchAllToPrimary()                 // 切回主模式
role.set(Role.PRIMARY)                        // 回滚状态
```

## 路径二：进程退出（GracefulShutdown）

### 完整步骤

```
SIGTERM
  ↓
GracefulShutdown.onApplicationEvent()
  Step 1: 暂停 Kafka 消费（pause，不释放 partition）
  Step 2: disruptorManager.shutdown()（排空所有消息，flush WAL+Redis）
  Step 3: kafkaRegistry.stop()（释放 partition，触发 rebalance）
  Step 4: leaderElection.startDeactivate() + completeDeactivate()
  Step 5: haService.persistRoleOnShutdown()  ← 清除 Redis PRIMARY 标志（已修复）
```

**与计划内切换的区别**：
- GracefulShutdown 先排空 Disruptor（Step 2），再停止 Kafka（Step 3），顺序更合理
- 不执行 createFinalSnapshot（进程即将退出，Snapshot 由下次 activate 时的 EventLog 回放补偿）
- Step 5 清除 Redis PRIMARY 标志（已修复隐患3），防止计划内停机后重启误自动激活

## 路径三：OOM 重启自动恢复

### 完整步骤

```
OOM → JVM 退出 → 进程重启
  ↓
HAService.init() → autoRecoverIfWasPrimary()
  读 Redis key: matching:ha:role:{instanceId}
  ├── 值为 "PRIMARY" → 自动调用 activate()
  │     STANDBY → ACTIVATING → PRIMARY
  │     失败时回滚到 STANDBY
  └── 无值或非 PRIMARY → 保持 STANDBY
```

**OOM 场景下 Redis PRIMARY 标志的状态**：

| OOM 发生时机 | Redis PRIMARY 标志 | 重启后行为 |
|------------|------------------|---------|
| activate 完成后 | 已写入 | ✅ 自动恢复为 PRIMARY |
| activate 过程中 | 未写入（activate 失败会回滚） | 保持 STANDBY |
| deactivate 完成后 | 已清除 | 保持 STANDBY |
| GracefulShutdown 完成后 | 已清除（Step 5 修复） | 保持 STANDBY |

---

## 三条路径对比

| 步骤 | 计划内切换（deactivate） | 进程退出（GracefulShutdown） | OOM 重启 |
|------|------------------------|---------------------------|---------|
| 状态机中间态 | ✅ DEACTIVATING | ❌ 无 | ✅ ACTIVATING |
| 停止 Kafka 消费 | ✅ stopConsumers() | ✅ pause + stop | N/A |
| 排空 Disruptor | ⚠️ 未显式等待 | ✅ disruptorManager.shutdown() | N/A |
| 创建最终 Snapshot | ✅ createFinalSnapshot() | ❌ 不执行 | N/A |
| 切换 EventLog 为只读 | ✅ switchAllToStandby() | ❌ 不执行 | N/A |
| 失败回滚 | ✅ 回滚到 PRIMARY | ❌ 无 | ✅ 回滚到 STANDBY |
| 清除 Redis PRIMARY 标志 | ✅ persistRole("STANDBY") | ✅ persistRoleOnShutdown()（已修复） | N/A |
| 数据恢复 | N/A | N/A | ✅ Snapshot + EventLog 回放 |

## 已修复的隐患

### ✅ 隐患3（已修复）：计划内停机后重启自动激活

**原问题**：`systemd stop` 触发 GracefulShutdown，但未清除 Redis PRIMARY 标志，重启后误自动激活

**修复方案**：GracefulShutdown Step 5 调用 `haService.persistRoleOnShutdown()` 清除标志

```java
// GracefulShutdown.java
haService.persistRoleOnShutdown();  // 删除 Redis role key
```

### ✅ 隐患4（已修复）：deactivate 失败后状态不一致

**原问题**：deactivate 中途失败，catch 块只打 log，实例进入半降级僵死状态

**修复方案**：失败时回滚到 PRIMARY，重新启动 Kafka 消费和 EventLog 主模式

```java
// HAService.deactivate() catch 块
kafkaConsumerStartupService.startConsumers();
eventLog.switchAllToPrimary();
role.set(Role.PRIMARY);
```

## 待优化项

### ~~隐患1~~（已澄清，非隐患）：deactivate 未等待 Disruptor 排空

**结论**：不需要等待，原因如下：
- DEACTIVATING 期间 Disruptor 里的存量消息分两类：
  - **已写入 EventLog**：从实例 activate 时通过 `loadFromEventLog` 补发增量 EventLog 给从实例，数据不丢
  - **未写入 EventLog**：对应 Kafka offset 未提交，从实例升主后会重消费这些消息重新撮合，数据不丢
- Snapshot 不完整没关系，从实例 activate 时 EventLog 增量回放会补齐
- **无需修改**

### ⚠️ 隐患2（待优化）：Snapshot 时序不一致

**现状**：多 symbol 的 Snapshot 非原子写入，各 symbol 恢复时间点可能不同

**影响**：低风险，有 EventLog 兜底，合约场景下跨 symbol 状态可能短暂不一致

**优化建议**：createFinalSnapshot 失败时告警，从实例 activate 对缺失 Snapshot 的 symbol 做全量 EventLog 回放

### 2. 性能优化策略

#### A. 计算优化
```java
// 核心优化：BigDecimal → long 运算
// 性能提升：3.57倍，内存节省：70-80%
public static long toLong(BigDecimal value) {
    return value.setScale(8, RoundingMode.HALF_UP)
                .movePointRight(8).longValueExact();
}
```

#### B. 内存优化
```java
// 对象池 + 零拷贝设计
- EntryPool: 订单对象复用
- Eclipse Collections: 原始类型集合
- Chronicle Queue: 内存映射文件
- ThreadLocal: 减少对象创建
```

#### C. 并发优化
```java
// 单Disruptor无锁并发框架
RingBuffer<MatchEvent> ringBuffer = RingBuffer.createMultiProducer(
    MatchEvent::new, 8192);
// 特点：
- 全局单Disruptor，所有交易对共享一个RingBuffer
- 单线程消费，避免锁竞争
- 单条处理模式：每条消息独立走完 撮合→EventLog→ack→异步IO
- asyncIoExecutor线程池处理Kafka发送/Redis写入等IO操作
```

## 高可用设计

### 1. 主从架构

#### 部署模式
```
机房A: Instance-A (PRIMARY)
  ├── 本地 NVMe SSD (Chronicle Queue EventLog)
  ├── Kafka 复制 EventLog → 从实例
  └── 定期写 Redis Snapshot (Protobuf)

机房B: Instance-B (STANDBY)
  ├── 本地 NVMe SSD (消费 Kafka 写入 EventLog)
  └── 切换时从 Redis 加载 Snapshot + 回放本地 EventLog
```

#### 故障转移流程
```bash

# 内部流程：停止Kafka消费 → 创建最终Snapshot写Redis → 切换EventLog → 更新状态

# 2. 激活备用实例
curl -X POST http://instance-b:8080/ops/ha/activate
# 内部流程：Redis读Snapshot → 回放本地EventLog增量 → 启动Kafka消费 → 更新状态

# 切换时间：10-30ms
```

### 2. 数据一致性保证

#### 事务性写入
```java
// Chronicle Queue事务机制
public long appendBatch(...) {
    long seq = allocateSequence();
    try {
        writeEvent(event);
        writeCommit(seq);
        updateCommittedSeq(seq);
        return seq;
    } catch (Exception e) {
        rollbackGlobalSeq(seq);
        throw e;
    }
}
```

#### 从实例复制写入
```java
// 保留主实例原始seq，避免主从seq不一致
public void appendReplicated(Event event) {
    writeEvent(event);  // 保留原始seq
    globalSeq.updateAndGet(cur -> Math.max(cur, event.getSeq()));
    committedSeq.updateAndGet(cur -> Math.max(cur, event.getSeq()));
}
```

#### Snapshot 机制
```java
// 定期60秒写Redis（Protobuf序列化），在asyncIoExecutor异步执行，不阻塞撮合线程
// Disruptor线程判断时间间隔 → 提交任务到asyncIoExecutor → 异步写Redis
```

## 高并发设计

### 1. 并发架构

#### 单Disruptor三阶段流水线
```
┌─────────────────────────────────────────┐
│     Kafka Consumer (多线程)              │ ← 消息接收层
├─────────────────────────────────────────┤
│     单Disruptor RingBuffer (8192)       │ ← 无锁队列（所有交易对共享）
├─────────────────────────────────────────┤
│     match-engine 单线程                  │ ← Phase 1: 撮合 + EventLog persist
│                                         │ ← Phase 2: ack Kafka offset
├─────────────────────────────────────────┤
│     asyncIoExecutor (4-16线程)          │ ← Phase 3: 复制/幂等/下游发送
└─────────────────────────────────────────┘
```

### 2. 并发优化技术

#### A. 单Disruptor + 单条处理
```java
// 全局单Disruptor，所有交易对共享
// Kafka多线程 → RingBuffer → 单线程消费（match-engine线程）
// 每条消息独立处理，三阶段流水线：
//   Phase 1: 撮合 + EventLog persist（关键路径，失败不ack）
//   Phase 2: ack Kafka offset（EventLog写成功即ack）
//   Phase 3: 异步IO（asyncIoExecutor线程池）
//     - Kafka复制EventLog给从实例
//     - Redis幂等标记
//     - 发送撮合结果给下游
```

#### B. 异步IO线程池
```java
// asyncIoExecutor (4-16线程) 处理所有非关键路径IO
// CallerRunsPolicy背压：线程池满时由Disruptor线程执行，自动限流
// Phase 3的所有操作都是best-effort，失败不影响撮合正确性
```

#### C. 异步深度发布
```java
// AsyncDepthPublisher 内部队列 + 背压处理
// 增量深度：每笔撮合后发布变化的价格级别
// 全量快照：定期5秒发布完整深度（asyncIoExecutor异步执行）
```

#### 硬件配置
```bash
# 服务器规格
CPU: Intel Xeon Gold 6248R (24核48线程)
内存: 128GB DDR4 ECC
存储: NVMe SSD 2TB (本地EventLog存储)
网络: 万兆以太网
```

#### 本地存储配置
```bash
# 创建本地EventLog目录
sudo mkdir -p /data/matching/eventlog-queue
sudo chown -R $USER:$USER /data/matching

# Kafka Topic创建（EventLog复制）
kafka-topics.sh --create \
  --topic MATCHING_EVENTLOG_SYNC \
  --partitions 10 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --config retention.ms=604800000 \
  --bootstrap-server kafka1:9092
```

### 2. 配置管理

#### 关键配置项

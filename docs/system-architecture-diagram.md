# 撮合系统架构设计文档

## 1. 系统概述

撮合系统是一个高性能、低延迟的金融交易撮合引擎，基于 JDK 21 + Spring Boot 3.5.7 + Chronicle Queue 构建，支持高并发订单处理和实时价格撮合。

### 1.1 核心特性
- **高性能**: TPS 100-500万，延迟 0.1-0.5ms
- **高可用**: 主从切换，故障恢复时间 10-30ms
- **数据一致性**: 事务性写入 + 检查点机制
- **实时性**: 毫秒级订单撮合和价格推送
- **可扩展**: 模块化设计，支持水平扩展

### 1.2 技术栈
- **运行时**: JDK 21 (LTS)
- **框架**: Spring Boot 3.5.7
- **消息队列**: Apache Kafka 2.8+
- **缓存**: Redis 6.0+
- **事件存储**: Chronicle Queue 5.24.4
- **序列化**: Protocol Buffers
- **监控**: Spring Boot Actuator + Micrometer

### 2.2 核心组件

#### 2.2.1 撮合引擎 (MatchEngine)
- **功能**: 核心订单撮合逻辑
- **特性**: 价格-时间优先算法
- **性能**: 单Disruptor单线程处理，所有交易对共享一个RingBuffer
- **数据结构**: 红黑树 + 双向链表
- **处理模式**: 单条处理三阶段流水线（撮合+EventLog → ack → 异步IO）

#### 2.2.2 事件日志 (EventLog)
- **实现**: Chronicle Queue（本地 NVMe SSD）
- **功能**: 事务性事件存储
- **特性**: 内存映射，零拷贝
- **性能**: 写入延迟 1-10μs
- **复制**: 主实例通过 Kafka 异步复制给从实例

#### 2.2.3 订单簿 (OrderBook)
- **功能**: 维护买卖盘深度
- **实现**: PriceLevelBook
- **优化**: 原始类型集合，避免装箱

#### 2.2.4 消息发布 (Publisher)
- **异步发布**: AsyncDepthPublisher
- **同步发布**: DepthPublisher
- **序列化**: Protocol Buffers

## 3. 数据流设计

### 3.1 服务职责分工

#### 3.1.1 撮合服务 (Matching Service)
- **核心职责**: 订单撮合、价格匹配
- **输入**: 来自trade-engine的下单/撤单请求
- **输出**: 成交结果、下单结果、撤单结果、深度数据
- **存储**: Chronicle Queue 本地事件日志 + Redis Snapshot
- **复制**: Kafka 异步复制 EventLog 到从实例

#### 3.1.2 交易引擎 (Trade Engine)
- **核心职责**: 资金管理、仓位管理、订单管理、结算管理
- **输入**: 用户交易请求
- **输出**: 向撮合服务发送下单/撤单信息
- **功能**: 风控检查、资金冻结/解冻、仓位计算

#### 3.1.3 清算服务 (Settlement Service)
- **核心职责**: 手续费计算
- **输入**: 消费撮合服务的成交结果、下单结果、撤单结果
- **输出**: 手续费账单、清算报告
- **功能**: 费率计算、手续费扣除

#### 3.1.4 行情服务 (Market Data Service)
- **核心职责**: 行情数据处理和推送
- **输入**: 消费撮合服务的成交信息、深度数据
- **输出**: K线数据、实时行情、深度信息推送给用户
- **功能**: K线聚合、行情计算、WebSocket推送

### 3.2 Kafka消息流向

```
Trade Engine → Kafka Topic: order-requests → Matching Service
                (下单/撤单信息)

Matching Service → Kafka Topic: trade-results    → Settlement Service (成交结果)
               → Kafka Topic: market-data     → Market Data Service (行情数据)
               → Kafka Topic: eventlog-sync   → Standby Instance   (EventLog复制)
```

### 3.3 订单处理流程

```
用户请求 → Trade Engine → 风控检查 → 资金/仓位管理
                ↓
         Kafka: order-requests
                ↓
         Matching Service → 撮合处理 → 本地Chronicle Queue + Redis Snapshot
                ↓                    ↓                      ↓
    Kafka: trade-results    Kafka: market-data    Kafka: eventlog-sync
                ↓                    ↓                      ↓
        Settlement Service    Market Data Service    Standby Instance
                ↓                    ↓              (写入本地EventLog)
           手续费计算           K线聚合 + 推送
```

### 3.4 事件流架构

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ Trade Engine│───▶│   撮合服务   │───▶│ Chronicle   │───▶│ Settlement  │
│ 下单/撤单    │    │ MatchEngine │    │ Queue(本地) │    │ 手续费计算  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                          │                   │                   │
                          ▼                   ▼                   ▼
                   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
                   │ 订单簿管理  │    │ Kafka复制   │    │ Market Data │
                   │ OrderBook   │    │eventlog-sync│    │ K线聚合    │
                   └─────────────┘    └─────────────┘    └─────────────┘
                          │                   │                   │
                          ▼                   ▼                   ▼
                   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
                   │   Redis     │    │ 从实例      │    │ WebSocket   │
                   │ Snapshot    │    │ 本地EventLog│    │ 实时推送    │
                   └─────────────┘    └─────────────┘    └─────────────┘
```

#### 事件流说明

1. **上游输入**: 从 trade-engine 接收下单/撤单信息
2. **撮合处理**: 撮合引擎处理订单，更新订单簿
3. **事件持久化**: Chronicle Queue 写入本地 NVMe SSD（延迟 1-10μs）
4. **Kafka复制**: 主实例通过 eventlog-sync topic 异步复制 EventLog 给从实例
5. **Snapshot**: 定期写入 Redis（Protobuf 序列化），用于快速恢复
6. **下游输出**: 成交结果发送给清算服务，深度数据发送给行情服务
7. **行情处理**: 行情服务聚合K线并推送给用户

### 3.5 数据一致性保证

1. **事务性写入**: Chronicle Queue 原子性操作
2. **Snapshot机制**: 定期写入 Redis（Protobuf 序列化）
3. **Kafka复制**: 主实例 EventLog 异步复制给从实例，按 symbolId 分区保证有序
4. **事件重放**: 故障恢复时从 Redis Snapshot + 本地 EventLog 增量重建状态
5. **幂等性**: L1 本地 LRU + L2 Redis 两级幂等控制

### 4.2 故障切换流程

1. **停止主实例**: `POST /ops/ha/deactivate`（创建最终 Snapshot 写 Redis）
2. **激活从实例**: `POST /ops/ha/activate`
3. **恢复数据**: Redis 读 Snapshot → 回放本地 EventLog 增量
4. **恢复完成**: 10-30ms 内完成切换

> 注意：不支持自动故障转移，需要运维手动操作

### 4.3 数据同步

- **EventLog复制**: 主实例通过 Kafka eventlog-sync topic 异步复制给从实例
- **Snapshot**: 定期写入 Redis（Protobuf 序列化），10 交易对约 10MB
- **恢复机制**: Redis Snapshot + 本地 EventLog 增量回放

## 5. 性能优化

### 5.1 计算优化

#### 5.1.1 数据类型优化
- **BigDecimal → long**: 性能提升 3.57倍
- **原始类型集合**: 避免装箱开销
- **对象复用**: 减少GC压力

#### 5.1.2 算法优化
- **红黑树**: O(log n) 查找性能
- **双向链表**: O(1) 插入删除
- **单条处理**: 每条消息独立走完三阶段，无批量聚合开销

### 5.2 I/O优化

#### 5.2.1 序列化优化
- **Protocol Buffers**: 性能提升 4.81倍
- **零拷贝**: 减少内存复制
- **单条序列化**: 每笔撮合结果独立序列化发送

#### 5.2.2 存储优化
- **内存映射**: Chronicle Queue 特性
- **顺序写入**: 最大化磁盘性能
- **压缩存储**: 节省磁盘空间

### 5.3 并发优化

#### 5.3.1 无锁设计
- **单Disruptor单线程撮合**: 全局单RingBuffer，所有交易对共享，避免锁竞争
- **单条处理**: 每条消息独立处理，无批量缓冲区
- **CAS操作**: 原子性更新

#### 5.3.2 异步处理
- **asyncIoExecutor**: 4-16线程池处理Phase 3非关键路径IO
- **事件驱动**: 响应式架构
- **CallerRunsPolicy**: 线程池满时自动背压

#### 分布式部署特点

1. **水平扩展**: 各层都支持水平扩展，可根据负载动态调整实例数量
2. **故障隔离**: 不同服务层之间相互独立，单点故障不影响整体系统
3. **负载分散**: 通过 UID Hash 和交易对分片，均匀分散负载
4. **高可用**: 每个撮合实例都有主从，支持快速故障切换
5. **数据一致性**: 通过 Chronicle Queue 本地存储 + Kafka 复制 + Redis Snapshot 保证数据一致性
6. **单实例多交易对**: 每个撮合实例通过单Disruptor处理多个交易对，单线程顺序处理保证一致性

## 14. 新架构流程详解

### 14.1 完整交易流程

根据新的架构设计，完整的交易流程如下：

```
用户层 → 系统网关 → 业务网关 → 交易引擎 → Kafka → 撮合服务 → Kafka → 清算服务 → 交易引擎
```

### 14.2 各层职责详解

#### 14.2.1 用户层 (Client Layer)
- **功能**: 用户交互界面
- **组件**: Web端、移动端、API客户端
- **职责**:
  - 用户登录认证
  - 交易指令输入
  - 实时数据展示
  - 交易结果反馈

#### 14.2.2 系统网关 Gateway
- **功能**: 系统入口和流量管控
- **核心组件**:
  - **负载均衡**: 分发请求到不同的业务网关实例
  - **认证授权**: JWT Token验证，用户权限检查
  - **限流熔断**: 防止系统过载，保护下游服务
  - **路由转发**: 根据请求类型路由到相应服务
- **技术实现**: Spring Cloud Gateway / Nginx

#### 14.2.3 业务网关 trade-broker
- **功能**: 业务逻辑预处理
- **核心组件**:
  - **订单验证**: 订单格式、参数合法性检查
  - **风控检查**: 用户限额、黑名单、异常行为检测
  - **参数校验**: 价格、数量、交易对有效性验证
  - **协议转换**: HTTP请求转换为内部消息格式
- **输出**: 标准化的交易指令发送给交易引擎

#### 14.2.4 交易引擎 trade-engine (UID Hash)
- **功能**: 用户状态管理和资金控制
- **分片策略**: 基于用户ID哈希分片，确保用户数据一致性
- **核心组件**:
  - **订单路由**: 根据UID Hash将订单路由到对应分片
  - **状态管理**: 维护用户订单状态、持仓信息
  - **持仓计算**: 实时计算用户可用余额、冻结资金
  - **资金检查**: 验证用户资金是否充足
- **数据存储**: 用户账户、订单状态、持仓信息

#### 14.2.5 Kafka 消息队列
- **功能**: 异步消息传递和系统解耦
- **主题设计**:
  - **订单主题**: 接收来自trade-engine的待撮合订单
  - **撮合主题**: 撮合相关的内部消息
  - **清算主题**: 撮合结果发送给清算服务
  - **通知主题**: 状态更新和用户通知
- **特性**: 高吞吐、持久化、分区并行处理

#### 14.2.6 撮合服务 (本系统)
- **功能**: 核心撮合逻辑处理
- **核心组件**:
  - **撮合引擎**: 价格-时间优先算法，订单匹配
  - **订单簿**: 维护买卖盘深度数据
  - **事件日志**: Chronicle Queue持久化所有事件
  - **深度发布**: 实时推送市场深度数据
- **特性**:
  - 单线程处理避免锁竞争
  - 内存映射文件提供极致性能
  - 事务性事件存储保证数据一致性

#### 14.2.7 清算服务
- **功能**: 交易后处理和资金清算
- **核心组件**:
  - **资金清算**: 处理买卖双方资金变动
  - **持仓更新**: 更新用户持仓数量
  - **手续费计算**: 计算并扣除交易手续费
  - **结算处理**: 完成交易最终结算
- **输出**: 清算结果发送回交易引擎

#### 14.2.8 交易引擎 trade-engine (回环)
- **功能**: 状态同步和用户通知
- **处理内容**:
  - **状态同步**: 根据清算结果更新订单状态
  - **持仓更新**: 更新用户账户持仓信息
  - **资金更新**: 更新用户可用资金和冻结资金
  - **通知推送**: 推送交易结果给用户

### 14.3 数据流转详解

#### 14.3.1 订单提交流程
1. **用户提交**: 用户在客户端提交买入/卖出订单
2. **网关处理**: 系统网关进行认证、限流、路由
3. **业务验证**: 业务网关验证订单参数、执行风控检查
4. **引擎处理**: 交易引擎检查用户资金、冻结相应金额
5. **消息发送**: 通过Kafka发送订单到撮合服务
6. **撮合处理**: 撮合服务执行订单匹配逻辑
7. **结果输出**: 撮合结果通过Kafka发送到清算服务
8. **清算处理**: 清算服务处理资金和持仓变动
9. **状态更新**: 交易引擎更新最终状态并通知用户

#### 14.3.2 消息流转模式
- **异步处理**: 各服务间通过Kafka异步通信
- **事件驱动**: 基于事件的响应式架构
- **最终一致性**: 通过事件重放保证数据最终一致
- **幂等处理**: 支持消息重复处理的幂等性

### 14.4 架构优势

#### 14.4.1 性能优势
- **水平扩展**: 各层都支持独立扩展
- **负载分散**: UID Hash分片均匀分散负载
- **异步处理**: 消息队列实现异步解耦
- **专业化**: 每个服务专注特定功能

#### 14.4.2 可靠性优势
- **故障隔离**: 单个服务故障不影响整体
- **数据一致性**: 事件日志保证数据不丢失
- **主从切换**: 撮合服务支持快速故障切换
- **消息持久化**: Kafka保证消息不丢失

#### 14.4.3 可维护性优势
- **服务解耦**: 各服务独立开发和部署
- **职责清晰**: 每个服务职责明确
- **技术栈灵活**: 不同服务可选择最适合的技术
- **监控完善**: 每个环节都有详细监控

### 14.5 关键设计决策

#### 14.5.1 为什么使用UID Hash分片？
- **数据一致性**: 同一用户的所有操作在同一分片
- **避免分布式事务**: 用户相关数据集中管理
- **负载均衡**: 用户分布相对均匀
- **扩展性**: 可根据用户增长动态调整分片

#### 14.5.2 为什么撮合服务独立？
- **性能要求**: 撮合需要极致的低延迟
- **算法复杂**: 撮合算法需要专门优化
- **状态管理**: 订单簿状态需要特殊处理
- **可靠性**: 撮合数据需要特殊的持久化机制

#### 14.5.3 为什么使用Kafka？
- **高吞吐**: 支持百万级TPS
- **持久化**: 消息不会丢失
- **分区**: 支持并行处理
- **回放**: 支持消息重放和故障恢复

## 🔑 关键发现

### 问题1：Sequential I/O 被分散

**当前状态**：
```
时间线：
t1: seq=1 (BTC) → Queue-BTC  ❌ 分散的磁盘写入
t2: seq=2 (ETH) → Queue-ETH  ❌ 跳过了BTC队列
t3: seq=3 (BTC) → Queue-BTC  ❌ 又回到BTC队列

结果：
- Page cache 效果差
- Disk seek 多（应该是0）
- Chronicle Queue 的 Sequential I/O 优势被破坏
```

**改进**：
```
时间线：
t1: seq=1 (BTC) → 统一Queue  ✅ 连续顺序写
t2: seq=2 (ETH) → 统一Queue  ✅ 同一流
t3: seq=3 (BTC) → 统一Queue  ✅ 同一流

结果：
- Page cache 完美命中
- Disk seek = 0
- Chronicle Queue 优势充分发挥
```

### 问题2：索引映射的维护成本

**瓶颈点**：
```
readEventBySeq(seq) {
    // 快速路径（命中）：微秒
    symbolId = seqToSymbolMap.get(seq);
    
    // 降级路径（miss）：毫秒！❌
    for (symbol in 30symbols) {
        // 扫描所有队列
    }
}

在 5k e/s 下：
- 索引容量 100,000 events
- 满一次需要 20 秒
- 清理后会产生miss
- Miss延迟 30-50ms → 正常延迟 1-2ms
- 阶梯延迟造成 P99 爆炸性增长
```

**改进**：
```
readEventBySeq(seq) {
    // 直接访问（O(1)）：纳秒
    return queue.readBySeq(seq);
}

恒定低延迟，无miss。
```

### 问题3：设计初衷已不存在

| 初衷 | 当初原因 | 现状 | 需要维持？ |
|-----|--------|------|----------|
| **业务隔离** | 交易对数据需要隔离 | 用户明确不需要 | ❌ NO |
| **HA独立性** | 故障转移更灵活 | HA按实例级别（不按交易对） | ❌ NO |
| **运维隔离** | 热门交易对独立备份 | 通过部署隔离解决，不需存储隔离 | ❌ NO |

**结论**：维持这三个能力的成本没有回报。

## ✅ 建议方案的优势

### 优势1：充分发挥技术栈优势
- Chronicle Queue 的核心优势是 **Sequential Disk I/O** 和 **O(1) Positioning**

## 🎓 技术背景知识

### Chronicle Queue 的设计初衷

Chronicle Queue 是为了解决以下问题而设计的：
1. **低延迟**：通过 Sequential Disk I/O（零随机访问）
2. **高吞吐**：通过 Memory-Mapped Files
3. **可持久化**：磁盘存储，支持审计、回放

按交易对分散存储 **违背** 了 Sequential I/O 的设计初衷。

### Disruptor 的角色

- Disruptor ✅ **保证了全局顺序性**
- 所有交易对事件通过单线程 Disruptor 处理
- Disruptor 已经解决了顺序问题，无需在存储层再分散

**结论**：既然 Disruptor 已保证顺序，无需在存储层维持分散隔离。
